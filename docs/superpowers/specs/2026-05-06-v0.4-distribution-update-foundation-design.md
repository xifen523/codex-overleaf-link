# v0.4 Distribution and Update Foundation Design

## Goal

Advance Codex Overleaf Link from the released `v0.3.0` baseline to an initial `v0.4.0` release candidate focused on distribution and update readiness.

`v0.4.0` is not a sync feature release. It keeps the `v0.3.0` full snapshot path, guarded browser writeback, undo flow, compile integration, and experimental OT warm mirror behavior intact. The release is complete when users and maintainers can install, update, diagnose version mismatches, build GitHub Release artifacts, and prepare a Chrome Web Store package without changing the core bridge architecture.

## Scope

In scope:

- Extension-to-native version and protocol compatibility checks.
- User-facing diagnostics for missing, old, future, or protocol-incompatible native hosts.
- Run gating so incompatible native hosts cannot receive execution or mirror mutation requests.
- Reproducible release artifact generation from the repository.
- GitHub Actions workflow for tag-based release artifacts.
- macOS installer updates for version-pinned install and clearer update guidance.
- Chrome Web Store preparation docs and extension package readiness.
- Version metadata updates for `0.4.0`.

Out of scope:

- Mandatory automatic native host updates.
- Windows or Linux native host installers.
- New sync architecture or changes to OT writeback behavior.
- Actual Chrome Web Store submission, developer account work, or review operations.
- Hosted backend, telemetry, or remote audit logging.

## Current Baseline

The current repository is at `v0.3.0`. `package.json`, `extension/manifest.json`, `README.md`, and `CHANGELOG.md` already describe the experimental OT warm mirror release. The native host already exposes `bridge.ping` with `version` and `protocolVersion`, and exposes `mirror.patchFiles` for the v0.3 OT warm mirror path.

The gap is that `protocolVersion` is informational only. The extension does not yet compare native host metadata against extension requirements, the popup treats any ping response as connected, and task execution paths do not consistently block incompatible native hosts before sending side-effecting requests.

## Architecture

`v0.4.0` adds a small compatibility layer shared by extension UI code and native-host tests. The extension sends its version and protocol requirements in `bridge.ping`. The native host returns version, protocol range, and capability metadata. Extension-side code classifies the response into a small set of statuses and exposes a single helper for UI and run gates.

Release tooling stays separate from runtime code. Build scripts create clean artifacts under `dist/releases/v0.4.0/`, excluding source-control metadata, tests, docs, temporary files, and platform-specific clutter. Release verification checks version consistency and required docs before artifacts are produced.

The installer remains explicit and user-driven. It supports installing a specific tag through `CODEX_OVERLEAF_REF=v0.4.0`, prints installed version details, and keeps Chrome extension reload as a manual step.

## Component 1: Compatibility Contract

Create `extension/src/shared/compatibility.js`.

Responsibilities:

- Define runtime compatibility constants:
  - extension protocol version: `1`
  - supported native protocol range: `min=1`, `max=1`
  - required native minimum version for v0.4: `0.4.0`
- Normalize semantic versions used by `package.json` and `manifest.json`.
- Build `bridge.ping` params from extension metadata.
- Evaluate native ping responses into a deterministic status:
  - `ok`
  - `native_missing`
  - `native_too_old`
  - `extension_too_old`
  - `protocol_unsupported`
  - `unknown_native`
  - `native_unhealthy`
- Return user-facing recovery metadata, including a canonical install command for the target release.
- Build `bridge.ping` params with this exact shape:

```js
{
  extensionVersion: '0.4.0',
  extensionProtocolVersion: 1,
  supportedNativeProtocol: {
    min: 1,
    max: 1
  },
  requiredCapabilities: [
    'bridgePing',
    'mirrorSync',
    'mirrorPatchFiles',
    'mirrorStatus',
    'codexRun',
    'codexCancel',
    'codexModels',
    'historyClearPlugin'
  ]
}
```

- Classify compatibility with this precedence:

| Condition | Status | User action |
|-----------|--------|-------------|
| Ping fails or native response is absent | `native_missing` | Install native host |
| Ping returns no native `version`, no `protocolVersion`, no `supportedProtocol`, or no `capabilities` | `native_too_old` | Reinstall/update native host |
| Native version is malformed | `native_too_old` | Reinstall/update native host |
| Native version is below `MIN_NATIVE_VERSION` | `native_too_old` | Reinstall/update native host |
| `capabilities` exists but any required capability is missing or not `true` | `native_too_old` | Reinstall/update native host |
| Extension version, when provided to native, is below native's optional `minExtensionVersion` | `extension_too_old` | Update extension |
| Native and extension protocol ranges do not overlap | `protocol_unsupported` | Update extension and native host together |
| Native response is compatible but `environment.codex.ok === false` | `native_unhealthy` | Fix local Codex environment |
| Native response is compatible and required environment is available | `ok` | Allow tasks |

`unknown_native` is reserved for future native responses that include version metadata but omit fields the current classifier does not recognize. Current v0.1-v0.3 native hosts must be treated as `native_too_old` so recovery guidance is deterministic.
If `environment.codex` is absent, compatibility stays `ok`; the panel should show environment details as unavailable and preserve existing fallback diagnostics. Only an explicit `environment.codex.ok === false` produces `native_unhealthy`.

Extend native-host `bridge.ping` response:

```json
{
  "host": "com.codex.overleaf",
  "platform": "darwin",
  "version": "0.4.0",
  "protocolVersion": 1,
  "supportedProtocol": { "min": 1, "max": 1 },
  "capabilities": {
    "bridgePing": true,
    "mirrorSync": true,
    "mirrorPatchFiles": true,
    "mirrorStatus": true,
    "codexRun": true,
    "codexCancel": true,
    "codexModels": true,
    "historyClearPlugin": true
  },
  "minExtensionVersion": "0.4.0",
  "environment": {}
}
```

`bridge.ping` remains backward-compatible as a method name. A v0.1-v0.3 native host that omits required v0.4 metadata such as `supportedProtocol` or `capabilities` is classified as `native_too_old`, not as compatible. `unknown_native` is only for future responses that include enough version/protocol metadata to prove they are not an old host, while still omitting fields this classifier understands.

Acceptance criteria:

- v0.4 extension plus v0.4 native host returns `ok`.
- v0.4 extension plus v0.3 native host is blocked with a native update message.
- Missing native host is distinguished from old native host.
- Protocol range mismatch is distinguished from semantic version mismatch.
- Compatibility helpers are covered by unit tests without relying on browser APIs.

## Component 2: User-Facing UI and Run Gates

Update panel diagnostics, popup status, and run preparation to use compatibility status.

Files:

- `extension/src/contentScript.js`
- `extension/src/popup.js`
- `extension/src/background.js`
- `extension/src/content/otWarmMirrorController.js`
- `extension/manifest.json`
- `extension/popup.html`
- `extension/src/shared/i18n.js`
- `extension/styles/panel.css`

Loading requirements:

- `extension/manifest.json` must load `src/shared/compatibility.js` before `src/content/nativeChannel.js` and `src/contentScript.js`.
- `extension/popup.html` must load `src/shared/compatibility.js` before `src/popup.js`.
- `extension/src/background.js` cannot rely on DOM script tags. Because the service worker script itself is `src/background.js`, it must load the helper with `importScripts('shared/compatibility.js')` or an equivalent path that resolves to `extension/src/shared/compatibility.js`. Tests must verify the service worker path can access the classifier.

Panel behavior:

- The "Check Local Connection" diagnostics call sends a compatibility-aware `bridge.ping`.
- The result shows extension version, native host version, protocol version, and status.
- If incompatible, the diagnostic card shows a short reason and a copyable install command.
- Existing environment diagnostics for Codex and LaTeX remain visible only after compatibility passes.

Popup behavior:

- The popup no longer treats any successful ping as enough.
- It reports:
  - native host missing
  - native host update required
  - extension update required
  - protocol mismatch
  - compatible connection
- It uses the same canonical install command as the panel.

Run gates:

- Before `codex.run`, `mirror.sync`, `mirror.patchFiles`, `task.run`, and `task.confirm`, the extension checks compatibility.
- Incompatible states prevent the request from being sent to the native host.
- The run transcript or diagnostics receives a concise failure event with next step guidance.
- `bridge.ping`, `mirror.status`, and diagnostics remain allowed so the user can recover.
- `codex.models` is allowed when the native host is reachable but incompatible, because model discovery is read-only and the existing static model fallback can still handle failures.
- `codex.cancel` is always allowed as best-effort when a native port exists, even if compatibility is not `ok`, because it may be needed to stop a run that started before a mismatch was detected. If delivery fails, the existing cancel failure event is used.
- `codex.history.clearPlugin` is blocked unless compatibility is `ok`, because it mutates plugin-local history.

Native method policy:

| Method | Compatibility policy |
|--------|----------------------|
| `bridge.ping` | Always allowed |
| `mirror.status` | Always allowed |
| `codex.models` | Allowed when native is reachable; fallback if it fails |
| `codex.cancel` | Always allowed best-effort |
| `codex.history.clearPlugin` | Requires `ok` |
| `mirror.sync` | Requires `ok` |
| `mirror.patchFiles` | Requires `ok` |
| `codex.run` | Requires `ok` |
| `task.run` | Requires `ok` |
| `task.confirm` | Requires `ok` |

Acceptance criteria:

- A v0.3 native runtime installed under the v0.4 extension produces a visible update-required UI and blocks execution.
- A compatible v0.4 runtime preserves current task behavior.
- Diagnostic technical details do not include Overleaf project content.

## Component 3: Release Artifact Build

Create:

- `scripts/build-release.mjs`
- `scripts/verify-release.mjs`

Add npm scripts:

```json
{
  "build:release": "node scripts/build-release.mjs",
  "verify:release": "node scripts/verify-release.mjs"
}
```

Artifacts are written to `dist/releases/v0.4.0/`:

```text
codex-overleaf-link-extension-v0.4.0.zip
codex-overleaf-native-host-v0.4.0.tar.gz
install.sh
uninstall-native-host.mjs
SHA256SUMS
release-manifest.json
```

Extension package rules:

- Include only files required by Chrome to load `extension/`.
- Exclude `.DS_Store`, tests, docs, `.git`, `dist`, temporary files, logs, and source maps unless explicitly required.
- Preserve the current manifest permissions unless Web Store prep discovers a required change.

Native package rules:

- Include `package.json`.
- Include `native-host/src`.
- Include `extension/src/shared` because native scripts depend on shared modules.
- Include `scripts/codex-json-agent.mjs`.
- Include installer and uninstaller scripts required to set up the native host.
- Exclude repository tests and unrelated docs.

Release manifest requirements:

- version
- git commit
- created timestamp
- artifact name, size, and SHA-256

Checksum rules:

- `release-manifest.json` records hashes for distributable payload artifacts only:
  - extension zip
  - native host tarball
  - copied installer scripts
- `SHA256SUMS` includes the same distributable payload artifacts and `release-manifest.json`.
- `SHA256SUMS` does not include itself.
- `release-manifest.json` does not include a hash of itself.
- `createdAt` is allowed to vary between builds; reproducibility means the same source tree and version produce the same artifact names, file inclusion rules, and verifiable checksums, not byte-identical manifests across wall-clock time.

Acceptance criteria:

- `npm run verify:release` fails if package, manifest, README badge, or changelog versions disagree.
- `npm run build:release` creates all expected artifacts.
- `SHA256SUMS` validates every generated artifact except `SHA256SUMS` itself.
- Release artifacts do not contain `.DS_Store`, `.git`, or test files.

## Component 4: GitHub Release Workflow

Create `.github/workflows/release.yml`.

Trigger:

```yaml
on:
  push:
    tags:
      - "v*.*.*"
```

Workflow:

1. Check out repository.
2. Set up Node.js 20.
3. Run `npm test`.
4. Run `npm run verify:release`.
5. Run `npm run build:release`.
6. Upload files from `dist/releases/<tag>/` to the GitHub Release.

Publishing behavior:

- Use GitHub's standard release action with repository `contents: write` permission.
- Create the release if it does not exist.
- Update or replace artifacts on an existing release for the same tag.
- Do not mark `v0.4.0` as prerelease.
- Do not mark the release as draft by default.
- Release title is the tag, for example `v0.4.0`.
- Release notes are generated from the matching `CHANGELOG.md` section by finding `## vX.Y.Z - ...` and taking all content until the next `## ` heading. If the section is missing or empty, the workflow fails before publishing artifacts. When rerun against an existing release for the same tag, the workflow replaces the release body with this extracted changelog section and replaces artifacts.

Acceptance criteria:

- The workflow is tag-driven and does not publish artifacts for ordinary branch pushes.
- Version mismatch fails before artifacts are uploaded.
- The workflow can be inspected without relying on local machine state.

## Component 5: macOS Installer and Update Flow

Update:

- `install.sh`
- `scripts/install-native-host.mjs`
- `scripts/uninstall-native-host.mjs`
- `README.md`
- relevant installer tests

Installer behavior:

- Continue to support macOS Bash 3.2.
- Continue to support `CODEX_OVERLEAF_REF`, `CODEX_OVERLEAF_REPO_URL`, `CODEX_OVERLEAF_INSTALL_DIR`, and `CODEX_OVERLEAF_EXTENSION_ID`.
- Make version-pinned installation the documented update recovery path:

```bash
CODEX_OVERLEAF_REF=v0.4.0 bash -c "$(curl -fsSL https://raw.githubusercontent.com/Ghqqqq/codex-overleaf-link/v0.4.0/install.sh)"
```

- Print source ref, package version, manifest path, bridge path, runtime root, extension path, and next steps.
- Tell users to reload the Chrome extension and refresh Overleaf after updating.
- Keep deterministic unpacked extension id support.
- Explicitly document that Web Store builds may need the Web Store extension id passed through `CODEX_OVERLEAF_EXTENSION_ID` or `--extension-id`.

Acceptance criteria:

- Installer tests cover the version-pinned path and user-facing output.
- README install/update instructions match the command shown in panel and popup compatibility guidance.
- Existing one-command install behavior remains available.

## Component 6: Chrome Web Store Preparation

Create docs:

- `docs/chrome-web-store/permissions.md`
- `docs/chrome-web-store/privacy.md`
- `docs/chrome-web-store/listing.md`
- `docs/chrome-web-store/release-checklist.md`

Permission document content:

- `nativeMessaging`: required to connect the extension to the local native host.
- `storage`: required for local extension preferences and session UI state.
- `https://www.overleaf.com/project/*` and `https://overleaf.com/project/*`: required to inject the panel only on Overleaf project pages, read project snapshots, and write accepted changes back through the browser.

Privacy document content:

- No hosted backend is added by this project.
- Project mirrors live under `~/.codex-overleaf/projects`.
- Plugin-isolated Codex state lives under `~/.codex-overleaf/codex-home`.
- Codex processing uses the user's configured local Codex CLI account.
- No default telemetry.
- Diagnostics exclude project content by default and ask users to redact logs in issue reports.

Listing document content:

- Short description.
- Detailed description.
- Feature bullets.
- Support and issue-report link.
- Required asset list:
  - existing 128 icon
  - screenshots
  - small promo image
  - optional marquee image

Acceptance criteria:

- Docs match the current manifest permissions and runtime data flow.
- Release checklist can be followed before Chrome Web Store submission.
- Docs explicitly state actual store submission is outside v0.4.

## Testing Strategy

Follow test-driven development for runtime behavior and release tooling.

Required automated checks:

- `npm test`
- `npm run verify:release`
- `npm run build:release`

New or updated tests:

- Compatibility helper unit tests.
- Native `bridge.ping` metadata tests.
- Popup compatibility guidance tests.
- Panel diagnostics compatibility guidance tests.
- Run gate tests for incompatible native host.
- Installer version-pinned update tests.
- Release artifact build and verify tests.
- Manifest metadata tests.

Manual smoke checklist:

- Install v0.4 from the version-pinned command.
- Load extension unpacked.
- Open an Overleaf project.
- Confirm diagnostics show extension and native host `0.4.0`.
- Simulate old native host by installing a v0.3 runtime and confirm tasks are blocked with update guidance.
- Run one ask-only task.
- Run one suggest-edit task and accept writeback.
- Confirm the experimental OT warm mirror control still exists and falls back as before.

## Definition of Done

`v0.4.0` initial version is complete when:

- `npm test` passes.
- `npm run verify:release` passes.
- `npm run build:release` produces every expected artifact.
- GitHub tag release workflow exists.
- Panel and popup clearly identify native host version mismatches and provide a fixed install command.
- Incompatible native hosts cannot receive execution or mirror mutation requests.
- README documents install, update, release artifact usage, and version mismatch recovery.
- Chrome Web Store prep docs exist and match the current permissions and data flow.
- No new sync architecture or automatic native host update behavior is introduced.
