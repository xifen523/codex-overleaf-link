# v0.6 Editor-Native Review UX Design

## Goal

Make Codex review and acceptance feel close to the Overleaf editor without changing the v0.5 safety model. v0.6 focuses on hunk-level review, editor jump/preview, large diff resilience, and review shortcuts while preserving the existing panel diff fallback.

## Context

The current v0.5 pipeline is stable:

- `native-host/src/codexSessionRunner.js` collects local mirror changes and attaches `diff` plus text `patches`.
- `native-host/src/diffEngine.js` builds file-level line hunks for display.
- `native-host/src/textPatch.js` builds guarded text patches for writeback.
- `extension/src/contentScript.js` renders file-level diff review and converts accepted file changes into operations.
- `extension/src/content/writebackController.js` builds `edit` operations from `syncChanges`.
- `extension/src/pageBridge.js` applies operations through Overleaf, with stale-write guards and save verification.
- `extension/src/page/overleafEditor.js` abstracts CodeMirror, textarea, and contenteditable read/write access.

v0.6 must extend this path rather than replace it.

## Scope

### In Scope

1. Hunk-level accept/reject for text `write` changes with valid patch ranges.
2. File-level accept/reject remains available for every change.
3. Jump from a panel hunk to the target Overleaf editor location.
4. Lightweight editor preview through selection/focus when CodeMirror internals are sufficient, with safe fallback when they are not.
5. Large diff rendering safeguards: collapsed/limited hunk rendering with explicit expansion.
6. Keyboard shortcuts scoped to the active diff review.
7. Tests for hunk selection, partial patch application, jump behavior, and release metadata readiness for v0.6.

### Out of Scope

1. No new sync protocol.
2. No writeback through Overleaf OT/realtime collaboration channels.
3. No bypass of stale-write guard, Reviewing/Track Changes gate, undo checkpoint logic, save verification, or auto compile gates.
4. No dependency or bundling step.
5. No hard dependency on Overleaf CodeMirror internals that lack fallback.
6. No binary asset upload; that remains v0.7.

## User Experience

When Codex returns local text edits in Suggest mode:

1. The panel shows each changed file.
2. Text write changes show individual review hunks when the change has safe local patches.
3. The user can accept or reject each hunk.
4. Accepting/rejecting a file still accepts/rejects all hunks in that file.
5. Selecting a hunk can jump the Overleaf editor to that location.
6. The user can use keyboard shortcuts only while the diff review has focus:
   - `j` / `k`: move hunk focus.
   - `a`: accept focused hunk.
   - `r`: reject focused hunk.
   - `Enter`: jump focused hunk to editor.
   - `Escape`: close keyboard focus back to normal panel flow.
7. When all required decisions are made, only accepted hunks are written.

When hunk review is unavailable:

- New files, deletes, unsupported patches, truncated large diffs, or full-file replacements fall back to file-level review.
- The panel should make this clear through concise status text; it must not imply that hunk-level selection is available when it is not.

## Data Model

`syncChanges` remains backward-compatible. v0.6 adds derived review metadata in the content layer and may later persist it from native code:

```js
{
  type: 'write',
  path: 'main.tex',
  previousContent: '...',
  content: '...',
  diff: [...],
  patches: [
    { from: 120, to: 135, expected: 'old', insert: 'new' }
  ],
  reviewHunks: [
    {
      id: 'main.tex:hunk:0',
      path: 'main.tex',
      patchIndexes: [0],
      startOffset: 120,
      endOffset: 135,
      startLine: 8,
      lineCount: 1,
      canApplyIndependently: true,
      truncated: false
    }
  ]
}
```

Initial implementation can derive `reviewHunks` from `patches` in `extension/src/content/reviewHunks.js`. Native-generated `reviewHunks` can be added later without changing panel semantics.

## Architecture

### Content Layer

Create `extension/src/content/reviewHunks.js` with pure helpers:

- `buildReviewModel(syncChanges)`: normalizes files and hunk records.
- `buildAcceptedSyncChanges(syncChanges, decisions)`: returns a filtered `syncChanges` array whose `patches` contain only accepted hunks.
- `summarizeReviewModel(model)`: produces counts for files, hunks, and fallback-only changes.
- `normalizeReviewDecisionKey(path, hunkId)`: stable keys for UI state.

`contentScript.js` should use these helpers to keep the main file from growing further. The existing `createDiffReviewElement()` remains the integration point but delegates model logic to the new helper.

### Page Bridge

Add a page method:

```js
jumpToPosition({ path, from, to })
```

Behavior:

1. Normalize and open `path` when it differs from the active file.
2. Wait for the target editor content to finish loading.
3. Focus/select `[from, to]` through `overleafEditor.js`.
4. Return `{ ok: true, method }` or a structured fallback failure.

### Editor Adapter

Extend `extension/src/page/overleafEditor.js` with:

- `focusActiveEditorRange(from, to)`.

Supported paths:

- CodeMirror view: `editorView.focus()` and `editorView.dispatch({ selection: { anchor, head } })`.
- Textarea: `focus()` and `setSelectionRange(from, to)`.
- Contenteditable fallback: focus only, return `selection-fallback`.

### Large Diff Behavior

Define conservative panel constants:

- `MAX_INITIAL_REVIEW_HUNKS = 20`.
- `MAX_INITIAL_HUNK_LINES = 80`.

The panel initially renders the first limited set and exposes an expansion button. Truncated hunks remain non-hunk-reviewable.

### Shortcuts

Keyboard shortcuts are attached only to the rendered diff review container. The handler must:

- Ignore events from inputs, textareas, selects, contenteditable nodes, and the Overleaf editor.
- Prevent default only for handled diff-review shortcuts.
- Never run when there is no active diff review.

## Safety

All accepted hunk writes still become normal `edit` operations with `patches`. That means:

- `CodexOverleafStaleGuard.checkOperationFreshness()` still verifies task-start content or exact patch ranges.
- `pageBridge.applyOperations()` still controls Reviewing/Editing mode.
- Undo generation still works from actual applied operations.
- Save verification, mirror refresh, and auto compile remain unchanged.

Partial hunk acceptance must never construct a `replaceAll` operation unless the original change was already fallback-only and accepted at file level.

## Tests

Required new/updated tests:

1. `test/reviewHunks.test.js`
   - builds one review hunk per patch.
   - filters accepted sync changes to selected patch indexes.
   - leaves create/delete changes file-level only.
   - reports truncated/fallback changes as non-hunk-reviewable.
2. `test/writebackController.test.js`
   - accepted partial hunks produce `edit.patches`, not `replaceAll`.
3. `test/pageBridgeStaleGuard.test.js`
   - `jumpToPosition` opens the target file and selects the expected range.
   - stale patch guard still blocks accepted hunk writes when expected text changed.
4. Source-level product tests
   - content script loads `reviewHunks.js` before `contentScript.js`.
   - diff review includes hunk accept/reject controls, jump controls, and keyboard handler.
   - large diff constants and expansion behavior are present.
5. Release tests
   - package and extension metadata can move to `0.6.0`.
   - release verifier no longer has stale v0.5-only expectations.

## Acceptance Criteria

1. A user can accept one hunk and reject another in the same file.
2. Only accepted hunks are written back.
3. A collaborator edit inside an accepted hunk range blocks writeback with the existing stale patch error.
4. A collaborator edit outside an accepted hunk range does not block that hunk.
5. Jump opens the correct file and focuses the expected editor range where supported.
6. Large diffs do not render all hunks eagerly.
7. Existing file-level diff review remains available.
8. `npm test` passes.
